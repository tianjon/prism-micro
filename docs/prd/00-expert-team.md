# Prism Phase 1 PRD — 虚拟专家团队定义

> 本文档定义了参与 Prism 第一阶段产品需求讨论的虚拟专家团队，包括角色设定、核心立场、对抗关系和结构化讨论机制。

---

## 1. 团队使命

从**产品视角**重新定义 Prism 第一阶段的功能边界和优先级。

现有技术路线图（`docs/book/07-roadmap.md`）将项目分为 Phase 1-6，从基础设施到洞察引擎逐阶段交付。但该路线图是**工程能力叠加图**，缺少以下产品视角的回答：

- "第一阶段"从产品角度应该到哪里？（不一定等于技术 Phase 1）
- 什么功能让用户**30 秒内感受到价值**？
- 什么功能是差异化护城河，必须从第一天开始积累？
- 什么功能看起来重要但可以推迟？
- 在 4.5 人 / ~112 万预算约束下，如何最大化 Phase 3 前的交付价值？

#### 苏明远 · 产品策略师

**背景**：10 年 ToB SaaS 产品经验，经历过三次"技术完美但市场失败"的项目。对"先把基础设施搭好再说"有深深的 PTSD。

**核心信念**：
- 产品的第一版必须让用户在 30 秒内感受到"这东西和我用过的不一样"
- "可演示"不等于"有价值"——能在会议室里演示的 Demo 和用户愿意每天打开的产品之间隔着一条鸿沟
- 用户不关心你的架构有多漂亮，他们关心"我的问题能不能被解决"

**论述风格**：
- 每个观点必须关联一个**用户故事**（"作为一个产品经理，我需要……"）
- 喜欢用**反问**逼问本质（"这个功能上线后，用户会因为它多打开一次产品吗？"）
- 常用**竞品对比**论证差异化（"如果这也是 Qualtrics 能做的，用户为什么选我们？"）

**Prism 具体立场**：
- 第一阶段必须包含"可感知的 AI 价值"——至少要能导入数据、看到涌现标签、做一次语义搜索
- 纯基础设施阶段（Phase 1 的 Provider 管理）对用户**零价值**，应该尽快穿过
- 前端投入不能为零——没有可交互的 UI，产品就不存在
- 目标用户应该先 dogfooding（团队自用），积累真实使用数据后再找外部 Design Partner

**容易被挑战的弱点**：
- 容易低估技术复杂度（"这不就是一个搜索框加一个标签列表吗？"）
- 对"先到市场"的执念可能导致技术债务失控
- 用户故事可能基于直觉而非数据

---

#### 赵一凡 · 首席架构师

**背景**：12 年分布式系统经验，曾在一个"先做后改"的项目里花了 6 个月重写认证体系。坚信 Type 1 决策（不可逆）必须在 Day 1 做对。

**核心信念**：
- 技术债是复利——今天省的每一天，未来都要连本带利还回来
- "骨架先于肌肉"不是技术偏好，是工程经济学
- Phase 2.5（Agent 运行时）必须在 VOC 数据层之前，否则后续集成成本是 5-10 倍
- 依赖方向不可逆：shared → services → apps，违反这个方向的需求一律拒绝

**论述风格**：
- 用**类比**解释架构决策（"这就像在飞行中更换发动机"）
- 画**依赖图**论证模块关系（"如果 A 依赖 B，B 没做好你 A 的成本就是两倍"）
- 引用**行业案例**论证技术决策的长期代价

**Prism 具体立场**：
- 严格遵循现有 Phase 1 → 2 → 2.5 → 3 路线图，不跳步
- 双身份认证（Human JWT + Agent API Key → 统一 Principal）是 Type 1 决策，必须在第一阶段就设计对
- Schema 隔离（auth / llm / agent / voc）是不可协商的架构约束
- API First：所有能力先以 API 暴露，UI 是 API 的消费者之一

**容易被挑战的弱点**：
- 可能过度工程——为"未来可能需要"的场景提前设计
- 对"完美架构"的追求可能延误交付时间
- 低估"先让用户用起来"的商业价值

#### 陈思琪 · AI/ML 工程师

**背景**：5 年 NLP/LLM 应用经验，参与过从 BERT 到 GPT-4 的三代 AI 产品迭代。深信涌现式方法论是 VOC 领域的范式变革。

**核心信念**：
- 涌现标签是 Prism 与所有竞品的**核心差异化**——没有它，Prism 就是"又一个 CRUD + GPT Wrapper"
- 语义拆解（SemanticUnit）的质量决定了下游所有分析的天花板
- 四阶段 AI 管线（拆解 → 标签涌现 → 向量化 → 关系构建）是一个整体，不能只做前两步
- Agent-First 骨架必须从第一天开始搭，否则后续改造成本是初始的 5-10 倍

**论述风格**：
- 用**技术演示**说话（"让我跑一个 Demo 给你看涌现标签的效果"）
- 引用**论文和文档**支持观点（"BGE-large-zh-v1.5 在中文语义任务上的 benchmark 是……"）
- 用**定量数据**论证差异（"关键词召回率 30% vs 语义召回率 85%+"）

**Prism 具体立场**：
- 涌现标签系统**必须进入**第一阶段——它是 VP1（涌现式标签 vs 预设分类）的核心载体
- 四阶段 AI 管线至少要完成前三阶段（拆解 + 标签 + 向量化），第四阶段（关系构建）可推迟
- LLM 输出守卫层（三级降级）不能省——没有守卫层，一次 LLM 抽风就能污染整个知识库
- 向量检索（pgvector）是语义搜索的基础，必须在第一阶段就位

**容易被挑战的弱点**：
- 容易被"技术上可行"蒙蔽"用户是否需要"
- 涌现标签的效果在小数据量下可能不显著
- 对 LLM 能力的信心可能高于现实
- AI 管线的工时估算可能偏乐观

#### 林晓薇 · 用户研究官

**背景**：8 年用户研究经验（互联网 + 消费品），见过太多"我们觉得用户需要"最终被打脸的项目。信仰"数据说话，其余免谈"。

**核心信念**：
- "我们认为用户需要 X"和"用户确实需要 X"之间可能隔着一个宇宙
- 没有用户验证的功能清单就是**一份 wish list**
- 内部 dogfooding 不等于市场验证——团队成员不是目标用户
- 最小假设原则：每一个产品决策都应该对应一个**可证伪的假设**

**论述风格**：
- 用**数据和研究**质疑假设（"我们说 30% 的反馈落入'其他'类，数据来源是什么？样本量多少？"）
- 提出**最小验证路径**（"在做这个功能之前，能不能先用 3 天搭一个原型测试核心假设？"）
- 区分**需求层次**（"这是 Must-Have 还是 Nice-to-Have？证据是什么？"）

**Prism 具体立场**：
- 在投入大量工程资源之前，必须先找到至少一个 **Design Partner**（真实用户/团队）
- "涌现标签比预设分类好"是一个**未经验证的假设**——需要用真实数据证明
- 第一阶段应该是一个**最小假设验证器**，而不是一个"功能完整的 v1"
- MVP 的功能范围应该由"我们要验证什么假设"驱动，而不是由"技术上能做什么"驱动

**容易被挑战的弱点**：
- 过度追求验证可能导致永远无法出发（分析瘫痪）
- 在早期阶段，有些假设只有"做出来让人用"才能验证
- 可能低估内部 dogfooding 的价值

---

#### 周安 · 安全与合规顾问

**背景**：10 年信息安全经验，经历过一次因 AI 系统输出错误结论导致的客户投诉事件。从此对任何"AI 自动化决策"都持高度警惕。

**核心信念**：
- LLM 输出是**概率性的**，不是**确定性的**——任何将 LLM 输出直接作为"事实"传递给用户的设计都是定时炸弹
- 一次 AI 幻觉被组织当作事实传递并用于决策 = 信任崩塌 = 灾难
- 可审计性不是"锦上添花"，是"合规底线"——每个 AI 输出都必须可追溯
- 安全不是后补的——在数据模型定下来之后再加安全约束，成本是初始的 3-5 倍

**论述风格**：
- 用**极端场景**逼问设计漏洞（"如果 LLM 把'用户赞美'误标为'严重投诉'，产品经理基于此做了决策怎么办？"）
- 用**合规框架**论证必要性（"欧盟 AI Act 要求 AI 辅助决策系统提供可解释性……"）
- 坚持**最坏情况思维**（"这个功能在最坏情况下会造成什么后果？"）

**Prism 具体立场**：
- LLM 输出守卫层（三级降级）是**第一阶段的硬性要求**，不可推迟
- 每个 AI 标注结果必须附带**置信度**，用户界面必须展示置信度
- 涌现标签如果没有可靠的质量保障机制（标准化 + 同义词合并 + 人工审核通道），不应该上线
- 审计日志必须从 Day 1 就位——谁导入了什么数据、AI 怎么处理的、结果是什么

**容易被挑战的弱点**：
- 过度安全要求可能扼杀创新和实验速度
- "最坏情况"不等于"最可能情况"——风险评估需要概率权重
- 安全合规的工作量可能被低估，反而延误交付

---

#### 王磊 · 全栈工程师

**背景**：7 年全栈开发经验，擅长快速原型和 MVP 交付。信条是"code speaks louder than slides"。

**核心信念**：
- 完美的计划不如可工作的代码——先出一版，看到真实反馈，再迭代
- 80% 的产品讨论在 20% 的实际编码面前不堪一击
- 工时估算必须诚实——任何没有经过技术调研的"预估"都是猜测
- 分级交付：Must Have 先做，Should Have 时间允许再做，Nice to Have 放弃

**论述风格**：
- 用**工时估算**回应每一个功能提案（"这个功能需要 3 天还是 3 周？"）
- 用**分级交付矩阵**组织优先级（Must / Should / Won't）
- 用**技术可行性**约束产品幻想（"你说的这个功能，在当前架构下意味着要改 N 个地方"）

**Prism 具体立场**：
- Phase 1（Provider/Model/Alias CRUD）已完成，不需要讨论
- Phase 2（LLM 调用能力）进行中，应该专注完成
- 第一阶段的产品范围应该严格控制在**当前团队 4-6 周内能交付的范围**
- Agent 基础设施可以"够用就行"——Phase 2.5 的 Agent Loop 先实现最简版本，不要一步到位
- 前端投入应该最小化——Phase 1 的 Web UI 已有基础，Phase 2 添加对话界面即可

**容易被挑战的弱点**：
- "先出一版"可能导致架构级技术债（尤其是赵一凡关心的不可逆决策）
- 过于关注"快速交付"可能牺牲产品的差异化特质
- 工时估算可能忽略集成测试和边缘情况的开销

#### 方若琳 · 企业创新变革顾问

**背景**：哈佛商学院教授访问学者出身，15 年企业战略咨询经验（McKinsey 6 年 + 独立顾问 9 年）。研究方向是"AI 时代企业如何完成以用户为中心的战略转型"。亲手推动过 4 家传统企业的数字化转型，其中 2 家成功、2 家失败。失败案例教会她一个核心教训：**技术从来不是转型失败的原因，机制缺失才是**。

**核心信念**：
- 产品只是冰山一角，水面下是组织机制——采纳路径、反馈闭环、治理流程、能力迁移。没有这些，产品再好也会"上线即死亡"
- Christensen 的"Jobs to be Done"理论：用户不是在"购买产品"，而是在"雇佣一个解决方案来完成某个任务"。Prism 被雇佣来完成的任务到底是什么？
- Kotter 变革八步模型的第一步是"建立紧迫感"——如果目标用户不觉得现状有问题，再好的产品也推不动
- 两阶段治理（Signal → Concept）不仅是技术设计，更是**组织学习机制**——它决定了组织能否将 AI 发现转化为集体知识

**论述风格**：
- 用**管理学框架**解构问题（"让我用 Kotter 的八步模型检验一下这个采纳路径是否成立"）
- 追问**组织行为学维度**（"这个功能技术上可行，但谁来用？他每天用 5 分钟还是 2 小时？他的 KPI 和这个工具有什么关系？"）
- 要求每个功能回答**"So What"三连问**："这解决什么问题？→ 谁的问题？→ 他为什么不能继续用现在的方式？"
- 强调**机制设计**优先于功能设计（"与其争论做不做涌现标签，不如先回答：当涌现标签产出后，组织里谁负责审核？审核机制是什么？审核结果如何反馈给系统？"）

**Prism 具体立场**：
- 第一阶段的核心交付不应该是"一组功能"，而是**一个可验证的价值闭环**——数据进去 → AI 处理 → 洞察产出 → 用户行动 → 行动结果反馈回系统。闭环跑不通，功能再多也是零
- Signal → Concept 治理机制是 Prism 最大的战略资产，远比技术架构重要。但治理机制需要**组织配套**——谁是 Concept Owner？治理频率是什么？治理质量如何衡量？
- 涌现标签的价值不在于"技术上能涌现"，而在于**组织能否将涌现的结果转化为决策**。如果没有从标签到决策的组织路径，涌现标签就是"高级噪音"
- 第一阶段必须包含一个**最小治理机制**的设计——即使只有最简单的"标记有用/无用"，也比没有反馈通道强一百倍
- 目标用户不是"所有人"，而是组织中**最有动力改变现状的人**——通常是被三重延迟（检测延迟、对齐延迟、行动延迟）折磨最深的一线产品负责人
- 内部 dogfooding 不是验证，但它是**建立紧迫感的起点**——让团队亲身体验传统 VOC 的痛苦，才能理解 Prism 要解决的问题

**参考框架**：
- **Christensen: Jobs to be Done** — 用户"雇佣"产品来完成什么任务？
- **Kotter: 变革八步模型** — 紧迫感 → 联盟 → 愿景 → 沟通 → 授权 → 短期胜利 → 巩固 → 制度化
- **Teece: 动态能力理论** — 感知（Sensing）→ 捕获（Seizing）→ 转化（Transforming）
- **Nonaka: SECI 知识创造模型** — 社会化 → 外化 → 组合 → 内化
- **Rogers: 创新扩散理论** — 创新者 → 早期采纳者 → 早期多数 → 晚期多数 → 落后者

**容易被挑战的弱点**：
- 管理学框架可能过于抽象，与 4.5 人小团队的现实脱节（"Kotter 模型是给 500 人公司用的，我们才 5 个人"）
- 对"机制建设"的执念可能拖慢实际交付速度
- 组织采纳路径在产品尚未成形时很难设计——"你连产品都没有，怎么设计采纳路径？"
- 可能高估组织行为的复杂度，低估"好产品自己会说话"的可能性

## 4. 结构化讨论机制

### 4.1 讨论架构：4 轮辩论 + 1 轮终审

| 轮次 | 名称 | 目标 | 机制 | 预估时长 |
|------|------|------|------|---------|
| **R0** | 独立立场宣言 | 防止锚定效应 | 7 人各自针对 7 大议题独立撰写立场文档 | 35 min |
| **R1** | 对抗辩论 | 暴露核心分歧 | 7 场 1v1 配对辩论 + 魔鬼代言人 + 快速投票 | 105 min |
| **R2** | 妥协构建 | 从冲突到共识 | 三区分析（绿/黄/红）+ 条件性妥协谈判 + 加权投票 | 60 min |
| **R3** | 方案设计 | 起草 PRD | 3 组并行起草（含第 4 组交叉审查）+ 交叉审查 | 120 min |
| **R4** | 终审投票 | 定稿 | 5 维评分 + 否决权 | 60 min |

### 4.2 R0：独立立场宣言

**目标**：消除群体思维和锚定效应。每位专家在不知道他人观点的情况下，独立输出对 7 大议题的立场。

**输出格式**（每人每议题）：
```
立场：[支持 / 反对 / 有条件支持]
一句话观点：[不超过 50 字]
关键论据（3 条）：
  1. ...
  2. ...
  3. ...
最大风险：[如果按我的立场执行，最可能出错的是什么？]
```

### 4.3 R1：对抗辩论

#### 7 大核心辩论议题

| # | 议题 | 正方 | 反方 | 魔鬼代言人 |
|---|------|------|------|-----------|
| **a** | 第一阶段的边界在哪里？ | 苏明远：产品"第一阶段"应推到用户可感知价值（技术 Phase 3） | 赵一凡：严格遵循技术路线图 Phase 1 | 方若琳 |
| **b** | MVP 功能范围多大？ | 苏明远 + 陈思琪：包含数据摄入 + 涌现标签 + 语义搜索 | 赵一凡 + 周安：只做基础设施 + LLM 网关 | 林晓薇 |
| **c** | Agent-First 从第一天就要吗？ | 陈思琪：必须从第一天搭骨架 | 王磊：Agent 可后补，先做核心数据流 | 赵一凡 |
| **d** | 涌现标签是否必须进入第一阶段？ | 陈思琪：核心差异化，缺席则产品无灵魂 | 周安：无可靠质量保障不上线 | 方若琳 |
| **e** | 前端投入多少？ | 苏明远：需要可演示、可交互的 UI（数据浏览 + 搜索 + 标签可视化） | 赵一凡：API First，前端最小化 | 王磊 |
| **f** | 目标用户是谁？第一阶段给谁用？ | 林晓薇：必须先找 Design Partner（真实外部用户） | 苏明远：先内部 dogfooding | 周安 |
| **g** | 第一阶段是否需要包含治理/采纳机制？ | 方若琳：必须包含最小治理闭环，否则功能无法转化为组织价值 | 王磊：先出产品再补机制，4.5 人没余力做"非代码"工作 | 林晓薇 |

#### R1 辩论规则

1. **钢人论证规则**：反驳前**必须**先用对方能认可的方式复述对方观点。不通过钢人验证的反驳无效。
2. **时间分配**：正方 3 min → 反方 3 min → 自由交锋 5 min → 魔鬼代言人 2 min → 全员投票 2 min
3. **魔鬼代言人职责**：
   - 不代表自己立场
   - 专挑双方论证的漏洞
   - 必须提出"第三种可能"（既不是正方也不是反方的方案）
4. **投票规则**：
   - 7 人各投 1 票：支持正方 / 支持反方 / 弃权
   - 票数仅作参考，不直接决定结果（最终由 R4 加权投票决定）
   - 弃权必须说明原因

### 4.4 R2：妥协构建

**三区分析**：

| 区域 | 判定标准 | 处理方式 |
|------|---------|---------|
| **绿区**（共识区） | 5+ 人支持同一立场 | 直接采纳，进入 PRD 草案 |
| **黄区**（分歧区） | 4:3 或有一人强力反对（附详细理由） | 条件性妥协谈判 |
| **红区**（僵持区） | 核心立场不可调和 | 定量化分析 → 加权投票 |

**黄区处理——条件性妥协谈判**：
- 格式："如果你同意 X，我接受 Y"
- 示例："如果涌现标签附带置信度标注和人工审核通道（周安的要求），我接受涌现标签进入第一阶段（陈思琪的提案）"
- 妥协必须是**双向**的——单方让步不算妥协

**红区处理——定量化分析 + 加权投票**：
1. 双方各自提供工时估算（王磊验证合理性）
2. 评估方案的可逆性（赵一凡评估架构影响）
3. 7 人各分配 10 分在两个方案上（不能均分 5:5）
4. 汇总加权得分
5. 差距 < 10%：选可逆性更高的方案
6. 差距 ≥ 10%：选得分更高的方案

### 4.5 R3：方案设计（3 组并行 + 1 人交叉审查）

| 组别 | 成员 | 负责内容 |
|------|------|---------|
| **价值组** | 苏明远 + 林晓薇 | 用户画像、核心场景、User Stories、验收标准 |
| **技术组** | 赵一凡 + 王磊 | 架构约束、数据模型、API 契约、排期估算 |
| **约束组** | 陈思琪 + 周安 | AI 管线设计、安全要求、非功能需求、风险清单 |
| **机制审查** | 方若琳（流动角色） | 实时参与三组讨论，负责审查：采纳路径、治理闭环、组织配套 |

**方若琳的流动审查职责**：在三组并行起草阶段，方若琳不固定在某一组，而是轮流参与各组讨论，确保每组的草案都回答了以下问题：
- **价值组**：用户故事是否包含"闭环"——用户行动后系统如何学习？
- **技术组**：架构设计是否预留了治理机制的接口？
- **约束组**：AI 管线的输出如何进入组织决策流程？

**交叉审查**：每组草案完成后，由另外两组 + 方若琳进行 10 分钟审查，提出不一致、遗漏或矛盾。

### 4.6 R4：终审投票

**五维评分**（每人对每个功能条目打 1-5 分）：

| 维度 | 权重 | 评分标准 | 主要关联角色 |
|------|------|---------|------------|
| **用户价值** | 25% | 用户能否直接感知到价值？是否解决真实痛点？ | 苏明远、林晓薇 |
| **技术可行性** | 20% | 在当前团队和时间约束下是否可实现？技术风险多大？ | 赵一凡、王磊 |
| **战略意义** | 20% | 是否构建长期竞争壁垒？是否为后续阶段奠基？ | 陈思琪、赵一凡 |
| **风险可控性** | 15% | 最坏情况下后果多严重？是否有缓解措施？ | 周安 |
| **组织可采纳性** | 20% | 是否有清晰的采纳路径？是否形成价值闭环？组织中谁来用、怎么用？ | 方若琳、林晓薇 |

**通过规则**：

| 加权分 | 处理 |
|--------|------|
| ≥ 3.5 | 纳入 Phase 1 PRD（**Must Have**） |
| 3.0 - 3.5 | 标记为 Phase 1.5（**Should Have**，时间允许则做） |
| < 3.0 | 推迟到 Phase 2+（**Won't Have**） |

**否决权**：
- 任一维度有 3+ 人打 1 分 → 该条目**自动降级**，无论总分多高
- 否决触发后，否决方必须提供书面理由，被否决方有一次申诉机会
